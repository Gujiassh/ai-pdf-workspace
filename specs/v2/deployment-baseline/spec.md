# 阶段 9 部署、日志与观测

## 状态

- 阶段：已完成
- 建立日期：2026-07-16

- 前置：V1 可用闭环与 V2-A Hybrid/RRF 检索质量已完成

## 问题

当前数据库、缓存和对象存储可由 Compose 启动，但完整应用部署、关键链路观测、备份恢复和安全入口仍未形成一套可重复验收的运行体系。仅能启动服务不足以证明故障可诊断、数据可恢复或公网入口可安全使用。

## 目标

在不改变业务 API、持久化模型和保存语义的前提下，建立可重复部署、可观测、可备份恢复、具备 HTTPS 安全入口，并能通过完整业务 smoke 验证的单机生产基线。

## 用户故事

### US1 一条部署命令启动完整系统（P1）

维护者准备好环境文件后，可以用一条编排命令启动 Postgres、Redis、MinIO、迁移、API、Worker 与 Web，不需要分别进入三个应用目录执行命令。

### US2 迁移完成后才接收流量（P1）

应用只能在数据库健康且迁移成功后启动。迁移失败时，API、Worker 和 Web 不应进入可用状态。

### US3 运行状态可判断（P2）

维护者可以从容器健康状态和单行日志判断 Web、API、Worker 与基础依赖是否已启动，以及迁移是否成功。

### US4 故障可定位（P1）

维护者可以从指标和单行日志判断请求失败率、provider 延迟、检索耗时、存储错误和任务积压，不需要进入数据库手工拼接状态。

### US5 数据可恢复（P1）

维护者可以生成包含数据库和对象存储的同批备份，并在显式确认后恢复到空环境；恢复后用户、文档对象和业务关系保持一致。

### US6 公网入口安全（P1）

访问者只通过反向代理进入 Web。生产域名自动启用 HTTPS，内部 API、数据库、缓存和对象存储 API 不直接暴露。

## 功能需求

- FR-001：Web、API 和 Worker 必须有可重复构建的应用镜像。
- FR-002：部署编排必须包含 Postgres、Redis、MinIO、migration、API、Worker 与 Web。
- FR-003：migration 必须在数据库健康后执行数据库升级，并在成功后退出。
- FR-004：API 和 Worker 必须等待 migration 成功；Web 必须等待 API 健康。
- FR-005：Postgres、Redis、MinIO、API 与 Web 必须定义健康检查。
- FR-006：部署需要的变量必须集中列出，并将密钥与宿主地址留给环境文件提供。
- FR-007：默认部署不得把 Web、数据库、Redis、MinIO API 或 API 业务端口直接暴露到公网；只发布 Caddy 入口，MinIO 控制台仅绑定本机回环地址。
- FR-008：部署编排必须使用具名持久卷保存 Postgres、Redis 与 MinIO 数据。
- FR-009：migration 与 API 必须使用同一个 API 镜像，避免迁移代码与运行代码版本不一致。
- FR-010：API 必须提供 Prometheus 文本指标，至少覆盖 HTTP 请求、provider、retrieval、storage 和 ingestion job 状态。
- FR-011：Worker 必须提供独立指标端口，并继续输出 grep-friendly 单行任务日志。
- FR-012：备份必须同时包含 PostgreSQL 逻辑备份、MinIO bucket 对象和校验清单。
- FR-013：恢复必须要求显式确认，先停止业务写入，再恢复数据库与对象存储，并可重复验证迁移 head。
- FR-014：反向代理必须成为唯一公开入口；Web、API、Postgres、Redis 和 MinIO API 不发布公网端口。
- FR-015：生产域名必须自动申请或加载 HTTPS 证书，并返回 HSTS、内容类型保护、frame deny 和 referrer policy 响应头。
- FR-016：部署验收必须通过上传、Worker 入库、Chat/citation、笔记和删除清理的真实业务 smoke。

## 非功能需求

- NFR-001：不改变现有业务 API、数据库字段含义、Chat/Citation 合同或保存语义。
- NFR-002：应用进程以非 root 用户运行，并通过容器停止信号执行现有优雅退出路径。
- NFR-003：镜像构建必须使用锁文件，部署配置必须能在提交前做静态解析验证。
- NFR-004：生产密钥不得写入版本库；示例值必须明确不可用于真实部署。
- NFR-005：指标 label 不得包含用户输入、文档名、对象 key、完整 URL 或其他高基数字段。
- NFR-006：备份与恢复脚本不得在日志中打印密码、token 或 API key。

## 成功标准

- SC-001：API、Worker 和 Web 镜像都能从干净构建上下文成功构建。
- SC-002：部署编排静态校验通过，服务依赖关系满足 FR-003 和 FR-004。
- SC-003：完整启动时 migration 成功退出，API 与 Web 变为 healthy，Worker 保持运行。
- SC-004：停止并重启部署后，数据库和对象存储数据仍存在。
- SC-005：部署文档给出准备、启动、状态检查、日志查看、停止和迁移重跑命令。
- SC-006：指标端点返回可解析的 Prometheus 文本，真实请求和 Worker 状态会更新对应指标。
- SC-007：在隔离环境完成一次备份、销毁数据卷、恢复和业务数据复查。
- SC-008：反向代理 smoke 返回 200 和约定安全头，内部服务端口保持未发布。
- SC-009：完整业务 smoke 中上传文档进入 ready、问答返回 citation、笔记成功保存、删除后文档和对象均消失。

## 非目标

- Kubernetes 或多节点高可用
- 云托管数据库与对象存储适配
- Prometheus、Grafana 或集中日志平台
- 多节点备份复制、增量备份和自动保留策略
- Kubernetes、云负载均衡或多区域灾备
